Task233 Unity

Unity 2022+ async scheduling primitives built for zero-GC hot paths, explicit time-unit delays, lightweight cancellation, WebGL-friendly single-thread execution, and measurable performance work.

Install

https://github.com/neko233-com/Task233-unity.git

For offline projects, create Assets/neko233/Task233 and copy only Runtime and Editor into it.

Why

Low overhead

Struct awaitables and pooled scheduler nodes avoid coroutine allocations for common frame and time waits after prewarm.

Unity native

PlayerLoop integration runs on Unity's main thread without requiring a scene object.

Benchmarkable

Performance tests are included, with an optional UniTask comparison assembly for direct speed and GC checks in under a minute.

Platform fit

No worker thread, unsafe, reflection emit, or dynamic code generation requirement for Unity 2022+ runtime platforms.

How It Works

Thin awaitables

Factory APIs return readonly structs. The default T233.Yield() path is a dedicated minimal awaitable with no cancellation or timing fields.

PlayerLoop queues

Each timing owns an array-backed continuation queue. Work is pushed into the queue and drained by a PlayerLoop runner on the Unity main thread.

Pooled delays

Frame and time waits use pooled delay nodes. Ready nodes transfer their retained cancellation handle to the continuation queue.

Single-thread cancel

Task233CancelSource is a struct handle backed by compact arrays, not a heap CancellationTokenSource.

Why Faster

Task233 wins these benchmarks by narrowing the hot path. UniTask is a broad async toolkit with many integration modes; Task233 is deliberately a small Unity PlayerLoop scheduler for common game waits. Less generality means fewer checks, fewer objects, and fewer layers on the measured paths.

Path Task233 design Performance effect
Yield() Dedicated default awaitable, no generic promise object, no timing conversion, no token plumbing. Factory creation stays close to returning default.
Frame delay Stores an integer frame counter in a pooled node and checks it during the selected PlayerLoop timing. Avoids coroutine allocation and avoids a general-purpose async source for the factory path.
Seconds / milliseconds APIs expose explicit units and store the target as a double timestamp. Milliseconds use a direct multiply to seconds. Avoids TimeSpan conversion and broad delay overload dispatch on the hot factory path.
Cancellation Uses integer handles, version checks, state bytes, and retain/release while work is pending. Avoids allocating CancellationTokenSource for each operation while keeping Cancel(); Dispose(); safe.
WebGL / single thread No worker thread dependency, no reflection emit, no unsafe requirement, no dynamic code generation. Same runtime shape works on Unity WebGL-style single-thread platforms.

Async methods can still allocate because C# generates state machines and captured closures. For the tightest paths, use static or cached continuations with T233.Post, prewarm expected capacities, and keep cancellation sources owner-scoped.

Cancellation Model

Task233 cancellation is designed for Unity ownership patterns. A UI panel, behaviour, or request owner can call Cancel() and immediately Dispose(). If scheduled work still holds the handle, Task233 retains the slot until that work observes cancellation, then releases it for reuse.

var cancel = T233.CreateCancelSource();

SearchAsync(cancel);

cancel.Cancel();
cancel.Dispose(); // safe while the awaited work is still pending

Business Coverage

Runtime tests cover async/await frame waits, explicit seconds and milliseconds APIs, low-diff migration aliases, cancellation exceptions, FIFO post order, nested workflows, owner cancellation, debounce, timeout cancellation, and the immediate Cancel(); Dispose(); pattern.

Quick Start

using Task233;
using UnityEngine;

public sealed class Example : MonoBehaviour
{
    private async void Start()
    {
        await T233.Yield();
        await T233.DelayFrames(3);
        await T233.DelaySeconds(0.25d);
        await T233.DelayMilliseconds(16);
        Debug.Log("continued on Unity main thread");
    }
}

Migration

Task233 keeps common UniTask and Task.Delay shapes as aliases, so hot-path migrations can be low-diff. New code should still prefer explicit-unit APIs.

Cancellation

var cancel = T233.CreateCancelSource();

try
{
    await T233.DelaySeconds(3, cancellation: cancel);
}
finally
{
    cancel.Dispose();
}
