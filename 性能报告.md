# Task233 性能报告

最后更新：2026-06-28

本报告记录 Task233 与 UniTask 的 Unity 2022.3.51f1 实测对比。当前口径：不做 warmup，单次 measurement，大循环记录总耗时，再倒推 `ns/op` 与 `ops/s`。本轮完整 EditMode 套件总耗时 23.86 秒。

## 测试环境

| 项目 | 值 |
| --- | --- |
| Unity | 2022.3.51f1 |
| 平台 | WindowsEditor / EditMode |
| Graphics API | Null (`-nographics`) |
| Scripting Backend | Mono2x |
| CPU | AMD Ryzen 9 7900X3D 12-Core Processor |
| 线程数 | 24 |
| 内存 | 64535 MB |
| OS | Windows 11 64bit |
| UniTask | 2.5.11 |
| UniTask 获取方式 | 下载 `2.5.11` 源码包并以 `file:` 本地 package 导入，不使用 git clone / UPM git 依赖 |
| 测试结果 | 28 passed / 0 failed / 0 skipped |
| 测试时间 | 2026-06-28 18:25:24-18:25:47 本机时间 |
| 输出文件 | `%TEMP%/Task233PerfRun20s/editmode-results.xml` |

| 项目 | 口径 |

## 全量对比总表

| 对比项 | Task233 | UniTask 2.5.11 | 当前结论 |
| --- | --- | --- | --- |
| 默认 yield factory | `T233.Yield()`，专用最小 struct awaitable | `UniTask.Yield()` | 已实测：Task233 1.03x 更快 |
| 指定 PlayerLoop yield | `T233.Yield(PlayerLoopTiming)` | `UniTask.Yield(PlayerLoopTiming)` | Task233 已支持，尚未单独实测 |
| 取消 yield | `T233.Yield(timing, Task233CancelSource)` | `UniTask.Yield(CancellationToken)` | 两者都支持；Task233 不依赖 `CancellationTokenSource` |
| 下一帧 | `T233.DelayFrames(1)` | `UniTask.NextFrame()` / `UniTask.DelayFrame(1)` | 已实测 `DelayFrame(1)` factory：Task233 34.7x 更快 |
| 多帧等待 | `T233.DelayFrames(n)` | `UniTask.DelayFrame(n)` | 已实测 factory；运行语义已覆盖 |
| 秒等待 | `T233.DelaySeconds(double)` | `UniTask.Delay(TimeSpan)` / `WaitForSeconds` | 已实测：Task233 46.5x 更快 |
| 毫秒等待 | `T233.DelayMilliseconds(int)` | `UniTask.Delay(int)` | 已实测：Task233 56.8x 更快 |
| 缩放时间 | 默认 scaled time | `DelayType.DeltaTime` 等 | Task233 已支持 scaled/unscaled；UniTask 支持更多 delay type |
| 非缩放时间 | `ignoreTimeScale: true` | `ignoreTimeScale` / `DelayType.UnscaledDeltaTime` | 两者都支持；Task233 API 单位更明确 |
| 固定更新等待 | `PlayerLoopTiming.FixedUpdate` | `WaitForFixedUpdate()` / timing 参数 | 两者都支持；Task233 用统一 timing 参数 |
| EndOfFrame | 可用 `PostLateUpdate` 近似调度 | `WaitForEndOfFrame`，含 coroutine runner overload | UniTask 功能更完整；Task233 尚未做专用 EndOfFrame API |
| 主线程 Post | `T233.Post(Action)` | `PlayerLoopHelper.AddContinuation` / UniTask 内部 continuation | Task233 对外直接提供；已测冷启动入队 |
| 取消源创建 | `T233.CreateCancelSource()` struct handle | `new CancellationTokenSource()` | 已实测：Task233 9.1x 更快 |
| Cancel 后立刻 Dispose | retain/release 延迟回收，挂起 await 仍能观察取消 | `CancellationTokenSource` 标准语义 | 两者都可安全表达；Task233 已覆盖业务测试 |
| Owner 生命周期取消 | 手动持有 `Task233CancelSource` | `GetCancellationTokenOnDestroy` 等扩展 | UniTask 生态更完整；Task233 更轻但需要显式管理 |
| Debounce | 手动 cancel 前一个 source | 常用 `CancellationTokenSource` 模式 | Task233 已有业务测试覆盖 |
| Timeout | 手动 delay 后 cancel source | `Timeout` / `TimeoutWithoutException` 扩展 | UniTask API 更丰富；Task233 低层能力已覆盖 |
| async/await 嵌套流程 | 支持 C# awaiter pattern | 支持 UniTask async method builder | 两者都支持；Task233 runtime tests 覆盖嵌套流程 |
| WhenAll / WhenAny | 尚未提供 | 已提供 | UniTask 胜；Task233 后续可做轻量组合器 |
| Forget / fire-and-forget | 尚未提供专用 API | 已提供 `Forget()` | UniTask 胜；Task233 当前建议用户自行捕获异常 |
| async LINQ | 不提供 | 大量 `IUniTaskAsyncEnumerable` LINQ | UniTask 胜；Task233 不把 LINQ 作为当前性能核心 |
| Unity async triggers | 不提供 | MonoBehaviour/UI/TextMeshPro triggers | UniTask 胜；Task233 当前聚焦 scheduler/runtime core |
| Coroutine bridge | 不提供 | `ToCoroutine` | UniTask 胜；Task233 不把 coroutine bridge 作为核心 |
| ThreadPool / background work | 不依赖 worker thread | 提供 `RunOnThreadPool` 等 | UniTask 功能更广；Task233 更适合 WebGL 单线程路径 |
| WebGL 单线程 | 无 worker thread 依赖 | 可用，但部分 ThreadPool API 在 WebGL 受限 | Task233 设计更窄，更容易保持单线程一致 |
| unsafe / dynamic code | 不需要 unsafe、reflection emit、动态代码 | 不以动态代码为核心 | 两者都适合 Unity；Task233 代码面更小 |
| 预热 | `T233.Prewarm(...)` | UniTask 内部池/PlayerLoop 初始化 | Task233 暴露容量预热，方便业务主动控 GC |
| Editor 预览 | `Tools > Task233 > Preview` | 无 Task233 仓库内对等窗口 | Task233 提供本包专用预览 |
| GitHub Pages 文档 | 已有 HTML 技术说明 | UniTask 有成熟 README/Docs | Task233 已补技术原理和测试报告 |
| 自动化性能测试 | 本仓库 GameCI + Unity Performance Testing | UniTask 自身另有测试体系 | Task233 仓库内已跑 28 项 EditMode |

说明：上表把“能直接实测的 hot path”和“功能/API 覆盖”分开看。Task233 当前不是 UniTask 全生态替代品；它的目标是把 Unity 主线程等待、调度、取消这些高频业务路径做得极薄、可测、低 GC。UniTask 在 async LINQ、Unity 事件触发器、组合器、协程桥接等广域能力上仍更完整。

## async/await 与业务场景覆盖

| 测试 | 覆盖内容 |
| --- | --- |
| `YieldContinuesOnNextPump` | `await T233.Yield()` continuation |
| `DelayFramesWaitsRequestedPumps` | 帧等待语义 |
| `DelayMillisecondsZeroCompletesSynchronously` | 毫秒 API 和同步完成 |
| `DelaySecondsZeroCompletesSynchronously` | 秒 API 和同步完成 |
| `CancelSourceStopsAwaitedWork` | await 中取消并抛出 `OperationCanceledException` |
| `PostRunsInFifoOrderForBusinessQueue` | 业务队列 FIFO 顺序 |
| `NestedAsyncWorkflowCompletesInOrder` | 嵌套 async/await 工作流 |
| `OwnerCancelPatternPreventsLateUiMutation` | UI/Owner 销毁后取消，避免 late mutation |
| `DebouncePatternKeepsOnlyLastRequest` | 搜索/输入 debounce，只保留最后请求 |
| `TimeoutPatternCancelsSlowWork` | timeout 取消慢任务 |

这组测试还覆盖了 `Cancel(); Dispose();` 立即调用时，挂起 await 仍能正确观察取消；Runtime 通过 cancellation retain/release 延迟回收句柄槽，避免 id 复用破坏旧等待。

## 平台适配目标

Task233 Runtime 不依赖 worker thread、反射生成、动态代码或 unsafe，调度基于 Unity PlayerLoop，目标覆盖 Unity 2022+ 的 Mono / IL2CPP / WebGL 单线程平台。当前 package 是 Unity-first 运行时；取消句柄和 awaitable 形状保持 .NET Standard 友好，后续可拆出传统 C# 通用 scheduler。

## 运行方式

先下载 UniTask 源码包到本地 `.perf/UniTask`：

```powershell
powershell -ExecutionPolicy Bypass -File Tools/PrepareUniTaskPackage.ps1 -Version 2.5.11
```

再运行 Unity 2022.3.51f1：

```powershell
"C:\Program Files\Unity\Hub\Editor\2022.3.51f1\Editor\Unity.exe" -batchmode -nographics -projectPath TestProject -runTests -testPlatform EditMode -testResults artifacts\editmode-results.xml
```

GitHub Actions 会下载 UniTask `2.5.11` 源码包、创建隔离测试工程，并运行同一套测试。仓库没有配置 Unity license secret 时，CI 会通过配置校验并跳过真实 Unity Editor 测试。
